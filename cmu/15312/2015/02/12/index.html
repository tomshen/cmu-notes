Erroneous ideas

  • interactive $\iff$ dynamic: There’s no coupling between interactive and dynamic languages. For instance, ML has a REPL, but is still a static language.
  • Bellman effect: Richard Bellman gave dynamic programming its name because he though “dynamic” sounded good. Just because something is “dynamic” doesn’t mean it’s better.
  • Anti-types: Types are an obstacle.
  • heterogeneity: [1, I, "one"] is ill-typed in ML. This is held up as an example of the limitations of static typing.

Technical issues

Dynamic languages are not opposed to static languages because dynamic languages $\subseteq$ static languages.

The crucial idea is that untyped languages are unityped languages $\approx$ dynamic languages. A dynamic language restricts attention to a single type. Bob Harper (unsurprisingly) thinks this is a bad idea.

Surface syntax can be very deceiving. What 2 + 3 means in ML is dramatically different from 2 + 3 in Python or + 2 3 in Lisp.

Thus, when we compile down to machine code, we’ll need to store type information as well. This costs space (to store types) and time (to check types).

In ML, 2 + 3 compiles down to a register holding 2, a register holding 3, and an adding operation. In a DPCF, 2 + 3 compiles down to memory holding num and 2, memory holding num and 3, and operations to not just add 2 and 3, but check the two types are the same and store that type with 5.

The formalisms behind a dynamic language force a less efficient implementation. And, this overhead (which is already substantial for such a simple language) increases even more as the language becomes more complex.

Type safety

One of the oft-touted benefits of dynamic languages is that your program will never fail to compile due to type checking.

Aside: ML’s error messages are terrible because it sets up a set of simultaneous equations and attempts to solve them with a system similar to Gaussian elimination. There’s no easy way to turn that into a good error message.

Theorem. DPCF is type-safe.

Proof.

  1. Preservation: trivial, since $d : \ok \mapsto d’ : \ok$ (“anything goes”)
  2. Progress: if $d : \ok$, then either
    • $d \m{ val}$
    • $d \m{ err}$
    • $\exists d’ : \ok . d \mapsto d’$

Addition in DPCF

Despite looking a lot like addition would in PCF,

  1. We have to check $\m{is_fun}\;p$ every recursive call of $p$ despite the fact that $p$ cannot be anything but a function. If we allowed unchecked function calls here, we’d have to allow it in situations where it would not be appropriate, like $\lambda f . f(\m{zero})$.
  2. We have to check $\m{is_num}\; p(y’)$ every recursive call for $\m{succ}$.
  3. We have to check $\m{is_num}\; y$ every recursive call for $\m{ifz}$.

Thus, this will be a lot slower than the same function in PCF.

Aside: If you wrote $\m{add}$ as a multi-argument function, we’re already introducing typing in the form of light pattern matching, since $\lambda(x,y)$ takes in $\langle \ok, \ok \rangle$ as an argument.

Conclusion

Structural operational semantics and type systems are extremely useful for understanding programming languages, especially when coupled with machine-checked proofs (e.g. Twelf).

Aside: A machine-checked proof ($\approx$ 30k lines of Twelf) was written in a couple weeks that proves the type safety of Standard ML.

It’s far too complicated to prove the type safety of dynamic languages.
