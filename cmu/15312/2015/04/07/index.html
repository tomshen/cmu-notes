Exceptions (continued)

We introduced classified terms, which are exceptions in ML.

This system is more powerful than ML’s exceptions. Here’s a way to implement it in ML (or any language with first-class functions and references), though it would probably be more efficient as a primitive.

signature CLSFD =
sig
    type clsfd
    type 'a class
    new : unit -> 'a class
    mk : 'a class -> 'a -> clsfd
    isofcls : 'a -> clsfd -> ('a -> 'b) -> 'b -> 'b
end

structure Clsfd :> CLSFD =
struct
    type clsfd = unit -> unit
    type 'a class = 'a option ref
    fun new () = ref NONE
    fun mk (r: 'a option ref) (x: 'a) = (fn () => r := SOME x)
    fun isofcls (r: 'a option ref) (c: unit -> unit) (f: 'a -> 'b) (x: 'b) =
        (
        r := NONE;
        c ();
        (case r of NONE => x | SOME y => f y)
        )
end

Thus, we can see as an abstract data type, with multiple possible implementations. One thing that must be true across all implementations is that classes must be dynamically generated.

Static Exceptions

Crary claims static exceptions (such as those in Java) don’t work. In our terminology, static exceptions are those that have classes that can be determined statically. In theory, this lets us reason about exceptions that a piece of code may raise.

However, if you have type abstraction, this doesn’t work. If new returns a static class, then if you have two classes with different 'as, they would still have the same classes, which breaks type safety. [My understanding of Crary’s explanation is questionable.]

Continuations

$$ \renewcommand\contt[1]{#1~\m{cont}} \renewcommand\letcc[3]{\m{letcc}_{#1}~#2~\m{in}~#3} \renewcommand\throw[3]{\m{throw}_{#1}~#2~\m{to}~#3} \renewcommand\cont[1]{\m{cont}(#1)} $$

We’re going to talk about first-class continuations. They give you the ability to capture the “sequel” of the evaluation of the program.

can be thought of as a function that does something, but never returns anything we care about.

Continuations let us have non-local exits. SML/NJ (but not Standard ML) has callcc: ('a cont -> 'a) -> 'a and throw: 'a * 'a cont -> 'b. C has setjmp and longjmp.

Consider a function that computes the product of a list of nats. We might want to exit early if any of them are 0, since there’s no point in computing the product any further.

λl : nat list .
    letcc exit in
        (fix product : nat list -> nat =
            λl . case l of
                nil => 1
            | 0 :: _ => throw 0 to exit
            | n :: t => n * product t)

We can use continuations for “time travel,” without the “paradoxes” that might result if had state.

datatype size = Large | Small of size cont

fun guessSize () = letcc k in Small(k)

(case s of
  Large => disp
| Small k => throw Large to k'); largedisp
compose (f : τ' -> τ) (k: τ cont) : τ' cont =
    letcc k'' in throw f (letcc k' in throw k' to k'') to k

Note that k'': τ' cont cont. We can use compose to implement coroutines in user space.

Comparing Exceptions and Continuations

Neither continuations nor exceptions are more powerful than the other. The two features are incomparable. You can implement exceptions with continuations and references.

Exceptions actually expand the computing power of a language, because you can implement a loop, which you can’t do with just continuations. In that sense, exceptions are “more powerful.”

On the other hand, with continuations, we expand the logical scope from constructive logic to classical logic. You can prove anything with loops, but that isn’t that useful from a logical perspective.
