Subtyping

We represents subtyping with $<:$.

The subsumption principle: if $\tau <: \tau’$, then a term of type $\tau’$ can be used anywhere a term of type $\tau’$ is expected.

Therefore, subtyping is reflexive and transitive. It may not be anti-symmetric.

$\tau$ is the subtype, and $\tau’$ is the supertype.

For instance, $\natt :> \intt$ means either

  1. the values of $\natt$ are values of type $\intt$ (inclusive), or
  2. the values of $\natt$ can be coerced into values of type $\intt$ (coercive).

There are three types of subtyping:

  1. subtyping on base types (e.g. $\natt <: \intt$)
  2. structural subtyping (general products and sums)
  3. nominal subtyping that you have to declare (e.g. $\m{Square} <: \m{Shape}$)

We “don’t like” nominal subtyping, because it’s just a veneer over some underlying relationship.

Width subtyping is subtyping on general products (records in ML) or general sums.

For general products, $\pair{i_1 \hookrightarrow \natt}{i_2 \hookrightarrow \m{bool}} <: \langle i_1 \hookrightarrow \natt \rangle$.

For general sums, $[i_1 \hookrightarrow \natt] <: [i_1 \hookrightarrow \natt, i_2 \hookrightarrow \m{bool}]$. Note the reversed ordering.

There are tradeoffs when it comes to implementation of subtyping. You can only have two of the following:

  1. cheap access
  2. inclusive interpretation
  3. unordered

Variance

Note that order matters for pairs, but labels matter for general products. $\pair{i_1 \hookrightarrow \natt}{i_2 \hookrightarrow \m{bool}} <: \pair{i_1 \hookrightarrow \intt}{i_2 \hookrightarrow \m{bool}}$ is depth subtyping.

Note the flipping for $\tau_1$. Functions are contravariant on the left, and covariant on the right. By default, things are covariant, and thus contravariance is more interesting. In fact, contravariance is the interesting thing about subtyping.

For instance, $\intt \to \natt <: \natt \to \natt$. This is because, if we expect to have a function that has a $\natt$ as input, being given a function that takes $\intt$ as input is fine. The reverse is not true, since we can’t pass an $\intt$ to a function that takes an $\natt$ as input.

References are neither covariant nor contravariant: they are invariant. Consider $\ref{\natt}$ and $\ref{\intt}$. If we expect an $\ref{\intt}$, but have a $\ref{\natt}$, and attempt to assign a $\intt$, it fails. If we expect a $\ref{\natt}$, but have a $\ref{\natt}$, and attempt to read it, it fails.

Three anecdotes on poor language design with respect to subtyping:

  1. Eiffel is a language with covariant functions. Naturally, this broke type safety. To patch around this, you need to do a global check to make sure your code isn’t horribly broken.
  2. Arrays are just “wider” references, and thus they should be invariant as well. In Java, arrays are covariant. Thus, Java has runtime checks, and you downcast every time you get something out of an array. This makes it really hard to have efficient arrays, and the compiler has to do a lot of extra work (dataflow analysis) to see if the downcasts can be omitted.
  3. There is a paper titled “Should Your Specification Language Be Typed?” by “a famous guy” (Leslie Lamport, of distributed systems and $\LaTeX$ fame). He assumed that references (called variables in his type theory) should be covariant (since everything should be covariant), and thus type safety is broken and useless. In essence, he showed that broken type theories are broken.

Bounded quantification

This language is called $F_{<:}$ (pronounced “F-sub”). Unfortunately, subtyping is undecidable. We can choose to just implement type checking anyways. Since the infinite-looping case only happens when we write a program that doesn’t type check, and since it’s pretty rare, we can sort of just be okay with that.
